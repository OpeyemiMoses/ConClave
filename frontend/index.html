The problem

A single reviewer, skimming under deadline, misses what four specialists working in parallel wouldn't. Architecture review misses the security hole. Security review misses the onboarding friction that keeps contributors away.

Nobody reads the whole repo before the decision gets made.

The conclave

Four agents explore the same repo at once, each with exactly one job. A fifth — the synthesis — cross-checks their findings against each other, resolves contradictions explicitly, and seals one verdict.

Ask again after a commit, and it diffs the change, re-examines only what's affected, and flags which of its own prior conclusions no longer hold.

The four

One narrow job each. That's the point.

Architecture

Structure & data flow

Maps components, entry points, and how a request actually moves through the system. Ignores security and dependency versions entirely — that's not its job.

"Centralized simulation loop — convenient for prototyping, but tightly coupled to three service modules."

Security

Secrets, auth, unsafe patterns

Greps for hardcoded credentials and unsafe patterns first, reads only what's flagged. Traces the auth flow — or the absence of one.

"Zero authentication on financial endpoints. Any client can read or modify any user's data."

Dependencies

Manifest & lockfile

Reads the manifest and lockfile, flags outdated majors, known-risky packages, and license concerns — without pretending to have a live CVE feed.

"4 direct dependencies, all permissively licensed. Light risk, lean surface."

Onboarding

Time to first contribution

Checks whether a new developer can get running from the README alone — setup steps, env vars, whether tests even run.

"No .env.example, no install steps. Half a day of reverse-engineering before a first contribution."
